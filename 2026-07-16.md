## Tablet RGB LEDs wired to the engine
The RK3576S panel's [[Tablet Wall Panel|status LED strip]] now reflects engine state. `/usr/local/bin/led-panel.py` — a **root Python daemon** (stdlib only) driven by `led-panel.service` (systemd, enabled → boot-persistent). Running as root meant the planned NOPASSWD sudoers wasn't needed.

- **Mapping (priority high→low):** EVAC → red, paging live → green, bell ringing → blue, engine unreachable → amber (fault). **Idle = dark** (Jamie removed the idle glow). Paging deliberately outranks ringing (a live mic is the salient state, and the bell is held by the page anyway).
- **All active states breathe** — a cosine pulse with an eased low-dwell and a 4% floor. **EVAC breathes faster (~1.6s)** to read as urgent; the others are ~4s (`BREATH_PERIOD_STATE` override).
- For a smooth 30fps it **writes the PWM `duty_cycle` sysfs directly** (init once, then modulate) instead of shelling out to `setcolor` each frame; the engine poll runs in a background thread so a slow request can't stutter the breath. Engine URL tries `.42:8095` then `.41:8095` (the piface Pi is DHCP and moves).
- **Verified on real hardware by reading PWM duty back** (can't see the LED remotely): idle = all off; EVAC = red channel sweeping 960784↔3921 over ~1.6s while G/B stay off = a clean breathing red. Source backed up to `Tablet\led\`.

## Tablet launcher — network panel (built autonomously)
Added network settings to the tablet's boot **launcher** (the "Choose server" screen), side by side with the server picker: see the IP, scan/connect WiFi, set a static IP or DHCP. Built and fully tested while Jamie was away (he authorised the connectivity risk).

**Why an agent:** the launcher is a static `file://` page with no backend, so it can't read or change the network. Added a small **root agent** — `/usr/local/bin/netpanel.py` (stdlib Python + nmcli), `netpanel.service`, bound to **127.0.0.1:8781** only. Endpoints: `GET /net`, `GET /wifi/scan`, `POST /ip`, `POST /wifi`, `POST /confirm`.

**Safety = auto-revert.** Every apply snapshots the current settings and rolls back after 30s unless confirmed (the engine's apply/confirm pattern), plus an immediate revert if `nmcli con up` fails its connectivity check. So a typo'd static IP or bad WiFi switch self-heals instead of stranding the panel. **Tested it for real:** applied a bad static IP (192.168.99.99, no gateway) with no confirm — the tablet dropped off the network, then **recovered to 192.168.1.40 in ~14s** on its own. Also verified static-same-IP apply + confirm (SSH held) and the read/scan paths.

**Kept the picker bulletproof:** the launcher stays `file://` so it never depends on the risky agent — if the agent's down it just shows "No network agent" and the server picker still works. The page talks to the agent over localhost; CORS handled with `ACAO:*` + `text/plain` POSTs to avoid preflight. **Confirmed the `file://`→127.0.0.1 fetch works in the tablet's own Chromium** — captured the live kiosk (populated with the real IP + WiFi list). Note: the tablet has no working screenshot tool (its ARM Chromium `--headless` crashes; only raw `xwd`), so I piped `xwd` through the office Pi's ImageMagick to get a PNG.

Tablet-specific site-glue, so not in the engine repo — backed up to `Tablet\net\` + `Tablet\kiosk\launcher.html`. Left the tablet on its normal DHCP/"Giraffe" config, launcher restored to the countdown behaviour.
