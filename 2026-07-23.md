# 2026-07-23

A long single-threaded day on [[Beyond Bell Commander]]'s Xilica bench rig (192.168.1.211, the Xilica POC — see [[Beyond Bell Commander]] for how it got repurposed). Started with a GPIO/UART investigation on the tablet (parked one, see [[Tablet Wall Panel]]), then pivoted entirely to a run of real Xilica feature work and bug fixes, each one verified live against the real Solaro QR1 (192.168.1.210:10007), not just in tests. **Committed and pushed** at the end of the day (`f40d7c1`, `tooling/beta`) — the whole day bundled into one commit given how interleaved the changes were across shared files (`service.py`, the console) rather than split retroactively into per-feature commits.

## Tablet UART lead (brief, before the pivot)
Ran the proper `dtc`/`libgpiod` toolchain (not the earlier hand-rolled parser) against the tablet's real device tree — confirms the earlier GPIO-scanning conclusion from [[Tablet Wall Panel]]: no relay/RS-485 nodes visible via GPIO, every claimed line is internal (touch/BT/WiFi/LCD/regulators). Real lead found instead: **two UARTs (ttyS6, ttyS9) are enabled, pin-muxed, and completely unclaimed** — likely the rear RS-485/RS-232 terminals per the vendor manual. Next legitimate step is a loopback test with a USB-serial adapter, not touched today. Full detail in [[Tablet Wall Panel]].

## Xilica Pink Noise: multi-zone + a real Insert Mixer bug
Added a Pink Noise speaker-test button to the Zones page (Setup → Sounds & Zones sibling, this is the live Zones tab). First version only drove `PinkNoiseZone{z}Mute` — on the user's prompt ("insert matrix also needs to be routed") traced the actual Solaro block wiring in `Bell System Demo.pjxml` and found pink noise has its own dedicated **"Pink Insert Mixer"** block (5 inputs: each zone's own signal + Pink Noise; 4 outputs straight into `Audio Output Level`, bypassing Exclusion/Duckers/Matrix entirely). Confirmed live on the real unit that `PinkNoiseZone{z}Level` ships **floored at -100dB regardless of mute state** — same "control reads back fine, no audio reaches the output" shape as the EVAC Ducker bug from [[2026-07-21]]. Fixed: driving both Mute and Level now.

Then, on request ("I should be able to select multiple zones at the same time"), reworked the whole feature from single-zone-exclusive to genuine multi-zone (any number of zones live together — each Insert Mixer cell is independent, unlike a bell/page which really does need one route). Added a **sig gen master volume slider** too (`PinkNoiseLevel` — the driver already had `set_source_gain` for this, nothing had ever called it).

## Media library bug: uploaded files invisible in the EVAC Sound dropdown
Reported as "not all files in media library are showing in sounds drop down." Root cause: uploading a file only ever pushed its matching Sound into the Setup page's **unsaved client-side draft** — since each top-level tab is a separate React component instance, navigating away before clicking Save silently discarded the draft Sound while the physical file stayed uploaded. Fixed server-side: the Sound is now created and persisted atomically with the upload itself (`_ensure_sound_for_file`), plus a `has_sound` flag + a "+ Add as sound" recovery button for files already stuck orphaned from before the fix. Found and fixed two real orphans live on the bench box in the process.

## Real crash bug: a 45-minute file OOM-killed the whole appliance
Follow-up report: "the system does not play the file is it too large?" It wasn't the 50MB upload cap (file was 37.6MB) — `MediaPlayer._load()` decodes an **entire** file into memory to play it, and a 45-minute mp3 blew up to ~3.8GB RAM. Confirmed via `dmesg`/`journalctl`: the kernel OOM-killer **SIGKILLed the whole `bellcommander` process three times in a row** as the user retried it, taking bells/paging/EVAC down each time. Fixed: upload-time duration guard (`soundfile.info()` on the header only, rejects anything over 10 minutes) — can't help files already in the library, so also cleaned up the live hazard (deleted the file, removed its dangling Sound, restored the "TEST" message that had been pointed at it).

## Speech intelligibility test button, next to Pink Noise
The deleted file turned out to be intentional — "purely for testing speech through the system," wanted trimmed to 10 min and made into a per-zone loop button beside Pink Noise. Recovered a surviving copy from `Documents\Audacity\`, trimmed **on the bench Pi itself** using a bounded `soundfile.read(frames=600*sr)` (can't repeat the OOM — never decodes past 10 min), downsampled to mono 22kHz, uploaded as **"Speech Test.wav"**.

Real architecture question: unlike Pink Noise (its own dedicated DSP-side Insert Mixer), nothing else has an independent per-zone routing path — Bells/Preamble/BGM/EVAC only route through the ordinary program system. Gave three real options (build a matching Insert Mixer in Designer / physically patch into the idle "Paging Local" crosspoint / reuse Preamble with zero new wiring). **Chose reuse-Preamble** — accepted trade-off: the speech test and a live page's preamble chime will interrupt each other, since they share the same DSP path. New `MediaPlayerConfig.speech_test_file` field + Setup → Media Player picker; `set_speech_test` loops the file via the media player *and* drives the Preamble crosspoint, unlike Pink Noise which is generated entirely inside the DSP and needs no media-player audio at all.

## RTA meters: fixed a real client-side delay
"The rta meters seem have a delay" — the backend's UDP-push meter cache was already fresh (confirmed ~10ms endpoint round-trip on the real Pi); the actual lag was the **client poll interval**: 500ms on the office console, 250ms on the wall panel, plus CSS transitions on top of both. Since the endpoint is a cheap in-memory read (no DSP round-trip per request), tightened both to 150ms poll + 80ms transition. Deployed to both the React console and the wall panel's own `panel.js`/`panel.css`.

## Fader unity fix: 100% now means 0dB, not +15dB
Two back-to-back requests, same root cause both times: "100% = +15 not 0db on xilica" (zone output level), then "the gain for the inputs need to be set aswell" (External Inputs faders). Both controls' real hardware ceiling genuinely is +15dB (confirmed against the vendor's control-tag export), so this wasn't a wrong-range bug — just an unwanted default, since 100% on an operator fader should mean "as loud as commissioned," not "boosted above unity." **AHM's and Atmosphere's drivers already did this correctly** — Xilica was the one outlier. New shared constant `FADER_UNITY_MAX_DB=0.0`, used by `set_zone_level` and `set_input_level` only; `set_source_gain` (sig gen volume, any future commissioning trim) still correctly spans the full +15dB range. Re-applied to every real zone and input on the bench Solaro so nothing was left at the stale boosted value.

## USB audio interface level — a real fix and a real hardware fault, both found live
"Is it possible to get more level out of the pi usb interface?" Checked ALSA directly rather than guessing: both C-Media "Unitek Y-247A" dongles (cards 2 and 3, the same generic hardware noted in earlier sessions) had their own **hardware output volume sitting at -23dB below maximum** — a setting completely separate from every software slider in the app, apparently just the driver's unmoved default since day one.

- **Card 2 (Bells/EVAC) — real fix.** Raised to max, persisted with `alsactl store` (otherwise `alsa-restore.service` would've silently reverted it on next reboot). Measured the actual effect on the real Solaro meter, before/after: Bells channel went from **~-88dB to ~-5dB** — roughly 80dB of real, audible improvement.
- **Card 3 (Preamble/BGM) — a different, unresolved hardware problem.** Even at full ALSA volume, Preamble shows **zero signal reaching the DSP at all** (pinned at the meter's absolute floor whether playing or not); BGM (the other output on the same dongle) shows only a very faint trace. Points to a bad cable, wrong input, or a partially faulty channel on that specific dongle — needs physical, on-site inspection, can't be fixed remotely. **Open item — check the cable from card 3's output into the Solaro's Preamble/BGM inputs next time someone's on the bench.**

## DAC hardware output level exposed as a real control
Follow-up to the USB dongle level fix above — until now, moving a DAC's own ALSA output level required SSH + `amixer`. Added `system.alsa_card_for_device_name()`/`alsa_dac_level()`/`set_alsa_dac_level()` and a Setup → Media Player slider per device, clearly separated from the per-channel software Volume column below it. Real gotcha caught live: these C-Media dongles' "Mic" control also exposes its own Playback volume (a monitor/sidetone level) — a naive "has pvolume + Playback" scan would have grabbed that instead of the real speaker output, so the picker checks a preferred-name allowlist first.

First pass, on request ("can i folderise the sound library?"): a free-text folder tag per file, grouped table. The user liked the idea but not the look, and asked for something modelled on the Calendar tab's own DayTemplate/Overlay colour chips instead — replaced with a real tag **registry** (name + colour, create/edit/delete), each file assigned via a coloured-pill `<select>`. That held up fine at first, but by 11 real files the *grouped-table* presentation itself (a separate mini-table per tag, full `<thead>` repeated each time) was "a mess" — flattened into **one table**, tag column styled as the pill itself rather than a chip-plus-dropdown pair, sound status collapsed to a single ✓. Confirmed "much better." Also added a confirm-before-delete on both the file and tag × buttons, after `STOP.wav` turned up genuinely deleted mid-session (recovered from a backup copy in `Documents\Audacity\`).

## Fader unity fix finished, and a font bug that wasn't
Walked the fader-unity fix (100% = 0dB, from the section above) all the way through every remaining control on request ("apply this logic to all faders") — the sig gen/Pink Noise volume slider and the per-zone Alert gain were the last holdouts on the raw +15dB ceiling. `GAIN_MAX_DB` deleted outright once nothing referenced it. Separately, "fonts on the evac/message page are missing something" turned out to be classic UTF-8-as-CP1252 mojibake in two EVAC message labels' em dashes (confirmed via a codepoint dump, not by trusting how the terminal printed it — which itself briefly re-mangled the "fixed" text mid-check and nearly caused a false alarm) — a live-data bug, not a font or code bug.

## Fader state now survives a restart — and a real dormant protocol bug found
"When the system is restarted volume sliders are not synced with xilica." Root cause: the appliance's own in-memory fader cache always started at a hardcoded 100%, since restarting the *service* never touches the DSP — real levels persisted there the whole time, just not reflected in the UI. Added the read-back half of every fader write (`db_to_level`/`raw_to_db`, `read_zone_level` etc.), called once at startup. **The very first live restart test scrambled the values between zones** — traced to a real, previously-dormant bug: the Solaro replies `OK`/`ERROR=<n>` to *every* command including a plain `SET`, and `_send_line` had never read that reply — it sat in the socket buffer and silently corrupted the next `GETRAW`. Fixed by draining and parsing the reply properly (same pattern `_get_raw` already used). Verified live twice: once reproducing the scramble, once confirming 4 zones + sig gen + one Alert fader all survived a restart exactly.

## Wall panel picks up config changes without a reboot
"When any panel ui changes are made the panel does not reflect them until after a restart... maybe a bounce on save." Added a `config_version` counter, bumped on every successful config save, exposed on `/api/status` (which every client already polls every 2s). The wall panel's `panel.js` now silently re-fetches the zone/macro/EVAC-button layout and messages when it sees the version change — no page reload (the kiosk has its own hard-won white-flash fix a reload risks retriggering), no device reboot. Verified via a clean server-side round trip (bumped the version, confirmed the new data was immediately queryable) rather than a live-browser screenshot, after an attempt to drive a genuinely isolated headless Edge instance for that turned out to silently merge into the real desktop browser instead (Windows/Edge single-instance quirk — worth remembering for next time).

## Manual clock set added next to the NTP card
"Add a manual time set option." For sites with no reliable path to an internet time server. `timedatectl` refuses a manual set while automatic sync is on, so setting the clock by hand now turns NTP off deliberately — re-applying a time server (the existing button) always turns it back on, closing the loop with no new control needed for that half. Caught a real live-hardware timing bug before shipping: back-to-back `timedatectl` calls raced ("Previous request is not finished, refusing.") — fixed with a short settle delay between them.

## Amp keep-alive tone: built, then changed from continuous to a pulse
"The amp goes to sleep and needs an audio signal to wake, takes about a second — can we send an inaudible tone to keep it awake, only 1hr before/after the last bell, make it an option." Confirmed scope first (all zones, level adjustable, window = 1hr either side of the whole school day, not just the last bell). Reused the Solaro's own Pink Noise generator/routing at an operator-set quiet level instead of the speaker test's full-unity level — same DSP controls, so it deliberately backs off whenever a manual Pink Noise test is running. Verified live: a direct call to the deployed driver moved one zone to exactly the expected dB for a given level, left the others floored, and reverted cleanly on stop.

Almost immediately followed by a refinement: "signal only needs to pulse every 15min, allow to set pulse time" — reworked from a continuous tone into a real pulse (brief burst, `threading.Timer`-scheduled end, configurable interval — default 15 min, pulse duration itself stays a fixed ~3s in code). Added a second status signal (`amp_keepalive_window_open`) since the old "is it on right now" flag is basically never caught by a UI polling every few seconds once the tone is only on for 3 seconds at a time.
