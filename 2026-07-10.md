### 🎛️ Second DSP — AtlasIED Atmosphere driver (validated live)

Proved the `Driver` abstraction: added a whole second DSP as **one file + a fake + tests**, no changes to the scheduler/UI/config. Bench unit = **AZM4 at 172.16.200.161** (web UI admin, third-party control on).

- **Protocol** — JSON-RPC 2.0 over TCP **:5321**, newline-delimited (`set`/`get`/`bmp`/`sub`); params `ZoneMute_N`, `ZoneGain_N` (dB), `ZoneSource_N` (routing), `PlayMessage_N` (bell trigger), `RecallScene_N`, `StopMessage`. Named params are **read-only**. Discovered the unit's whole config by probing the protocol (zones Warehouse/New Zone, sources BGM, messages Alarm/STOP/Chime).
- **Two-way commissioning** — Setup → Sounds & Zones is now **DSP-aware**: **↺ Load from Atlas** (Pull names in) and an **Atlas Push card** (Bell Commander = master → generates the "create these on the Atlas" build sheet + **Verify** the unit matches, since names can't be pushed).
- **Group → zone routing (Option A)** — the Atlas can't steer a message's zones per-trigger (confirmed: no such param), so **each Atlas Message = a sound + its zones**; **Generate bell messages** derives one per `(sound, group)` the timetable uses; the driver resolves `trigger_bell(sound,group)` → the right message. (Option C, appliance-as-Dante-source + `ZoneSource`, is the future path for **music bells**.)
