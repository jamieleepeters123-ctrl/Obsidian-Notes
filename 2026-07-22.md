## Zones tab: real level controls
The Setup web UI's Zones tab had zone Mute but no Level control at all — and no way to trim an individual source's level per zone. Built both:
- **Per-zone output level** — a slider next to the existing Mute toggle, backed by a new `set_zone_level`/`/api/zone-level` (the driver method already existed for every DSP; only the service/API layer was missing).
- **Per-zone, per-source level matrix** — click a zone's ▸ to expand a 5-slider panel for Bells/Alert/Preamble/Paging Mic/Paging Local. New driver capability `set_zone_source_level` (crosspoint `Level` controls for the first four, `Zone{z}AlertGain` for Alert — a real per-zone gain stage, just not a crosspoint). **Spare deliberately excluded** — confirmed no per-zone control exists for it on this DSP, only the shared master already on External Inputs; a fake per-zone slider for it would've been misleading. Both decisions (Spare handling, expandable-row layout vs one big matrix) were explicit user choices.

Verified the whole write path against the real Solaro: set fader position 0.5 → confirmed `Zone1BellsLevel` read back `-25.0dB` on the DSP directly, then restored to unity.

## Fader taper
Direct user feedback: "the linearity of the faders — only the top 30% is good." Root cause: `level_to_db()`'s straight linear map put most of a fader's useful travel on deep attenuation nobody drags to (70% position was already `-30dB`, near-inaudible). Replaced with a squared taper (`dB = hi − (hi−lo)×(1−level)²`) — endpoints unchanged, but 70% now lands around `-9dB`. This is the single conversion point every fader in the system goes through (zone level, the new per-source matrix, External Inputs, the tablet panel's own faders), so one fix covered everywhere. Applied the same taper to the AHM and Atmosphere drivers too, on request — AHM's is unverified against real hardware (no documented MIDI-to-dB curve for that unit), Atmosphere's is a direct dB formula, same confidence as Xilica's.

## Real UDP meters, not polling
User pointed out RTA meters weren't showing anything and asked whether the engine was actually using UDP with a keep-alive — it wasn't; despite `subscribe_meter`/UDP infrastructure existing in the driver, meters were (and always had been) implemented as synchronous `GETRAW` polling, and neither `subscribe_meter` nor `ping()`/keep-alive were ever called from anywhere. Rebuilt properly per the vendor's own third-party control docs (`support.xilica.com/en/articles/3940994`): `SUBSCRIBE <ctrl> "UDP"` on connect, a background thread receiving the broadcast on :10008, automatic re-subscribe on any TCP reconnect (the docs are explicit that a lapsed session drops every subscription), and a 30s `KEEPALIVE` (half the vendor's 60s budget).

First implementation still returned nulls — captured the actual raw UDP traffic directly (`#AlertPlayerMeter=-88.50\r`) and found the push payload is **plain decimal dB**, not `GETRAW`'s dB×1000 raw integer, which the parser had assumed. Fixed; confirmed live with values genuinely changing between checks with zero polling on the engine's side.

Along the way, verifying the live meters endpoint surfaced one more real bug: an input literally named **"Paging Aux"** (not "Paging Local") was silently unmapped — the earlier word-matching only recognised "local". Added `{"paging","aux"}` as a second recognised phrase mapping to the same Solaro source.
