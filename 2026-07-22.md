## Fader taper
Direct user feedback: "the linearity of the faders ‚Äî only the top 30% is good." Root cause: `level_to_db()`'s straight linear map put most of a fader's useful travel on deep attenuation nobody drags to (70% position was already `-30dB`, near-inaudible). Replaced with a squared taper (`dB = hi ‚àí (hi‚àílo)√ó(1‚àílevel)¬≤`) ‚Äî endpoints unchanged, but 70% now lands around `-9dB`. This is the single conversion point every fader in the system goes through (zone level, the new per-source matrix, External Inputs, the tablet panel's own faders), so one fix covered everywhere. Applied the same taper to the AHM and Atmosphere drivers too, on request ‚Äî AHM's is unverified against real hardware (no documented MIDI-to-dB curve for that unit), Atmosphere's is a direct dB formula, same confidence as Xilica's.

## Real UDP meters, not polling
User pointed out RTA meters weren't showing anything and asked whether the engine was actually using UDP with a keep-alive ‚Äî it wasn't; despite `subscribe_meter`/UDP infrastructure existing in the driver, meters were (and always had been) implemented as synchronous `GETRAW` polling, and neither `subscribe_meter` nor `ping()`/keep-alive were ever called from anywhere. Rebuilt properly per the vendor's own third-party control docs (`support.xilica.com/en/articles/3940994`): `SUBSCRIBE <ctrl> "UDP"` on connect, a background thread receiving the broadcast on :10008, automatic re-subscribe on any TCP reconnect (the docs are explicit that a lapsed session drops every subscription), and a 30s `KEEPALIVE` (half the vendor's 60s budget).

First implementation still returned nulls ‚Äî captured the actual raw UDP traffic directly (`#AlertPlayerMeter=-88.50\r`) and found the push payload is **plain decimal dB**, not `GETRAW`'s dB√ó1000 raw integer, which the parser had assumed. Fixed; confirmed live with values genuinely changing between checks with zero polling on the engine's side.

Along the way, verifying the live meters endpoint surfaced one more real bug: an input literally named **"Paging Aux"** (not "Paging Local") was silently unmapped ‚Äî the earlier word-matching only recognised "local". Added `{"paging","aux"}` as a second recognised phrase mapping to the same Solaro source.

## Second wave: Zones tab Inputs, panel fixes, a real NIC bug
After the first backup pass, more real work landed:
- **Zones tab gained an "Inputs" card** ‚Äî the same master level/mute/RTA-meter faders the tablet panel's own External Inputs page has, for every commissioned input, now also reachable from the office console. In response, the **panel's own External Inputs page was trimmed to just the Paging Aux fader** ‚Äî everything else is a commissioning trim, no longer worth duplicating on the touchscreen now both places aren't needed.
- **Panel not showing as connected, and brightness not responding, turned out to be the same root cause**: `netpanel.py`'s heartbeat thread was dying permanently on a transient network error at boot (an unguarded `discover_engines()` call inside an otherwise fault-tolerant loop). Fixed and verified live ‚Äî `/api/panels` shows the tablet online again, brightness round-tripped correctly.
- **Panel brightness field in the Setup UI restyled** to match the local-display slider it sits next to (was a plain number input).
- **Panel dark/light mode added**, same palette as the office console's own themes. Emoji icons (‚òÄÔ∏è/üåô) didn't render reliably on the tablet's font stack (moon showed as a tofu box) ‚Äî switched to inline SVG, matching every other icon in the panel. Both confirmed via real screenshots.
- **A genuine NIC hardware bug, unrelated to any of the above**: wired Ethernet dropped and wouldn't reconnect (stuck retrying DHCP forever, though the physical link itself was fine). `dmesg` showed the immediate cause ‚Äî a PCIe power-management fault: the r8168 NIC failed to wake cleanly from D3hot back to D0, forcing a hard reset that DHCP couldn't recover from on its own. First fix (a udev rule forcing `power/control=on`, disabling *runtime* PM idle-suspend) was real but incomplete ‚Äî it happened again straight away. The user correctly identified the actual trigger: the physical power button. `dmesg` confirmed a genuine kernel suspend-to-RAM (`PM: suspend entry (deep)`) every time, traced to a standalone vendor script (`/usr/bin/power-key.sh`) that suspends the kernel directly on every short press, completely bypassing `systemd-logind`. Real fix: changed the button to blank the screen instead of suspending the system at all ‚Äî a wall-mounted paging station should never actually suspend, regardless of what triggers it. Full details in [[Tablet Wall Panel]].
