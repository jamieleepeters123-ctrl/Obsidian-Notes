## Bell Commander — general bug hunt + Xilica goes real

### Morning: general bug audit (not Xilica-specific)
Ran a fork-based audit of the whole engine looking for the same class of bug as the `rang_total` fix from the previous session. Found and fixed two real ones:
- **Today page "rung" count was a lifetime total, not today's** — `AuditLog.count()` had no date filter; `since` param added, `status()` passes site-local midnight.
- **Skip Next only tracked bell time, not which bell** — two bells at the same time, different groups (an explicitly supported layering pattern) meant skipping one silently skipped/mislabeled the other too. Fixed: skip identity keyed on `(time, group)`; a dedicated `_skipped` set so the timeline shows "skipped" correctly even after the bell's time has passed (tick() discards the matching `skip_times` entry once used).
- Small UI fixes: Next-bell countdown rolls into hours past 60 min; Sounds & Zones Media file column only shows when the media player is enabled; a "who plays the audio" badge added to that page.
- **System page: driver picker added** (`GET/PUT /api/system/driver`) — pick AHM/Atlas/Xilica + address from the console instead of hand-editing `device.json`. Found + fixed a real footgun while building it: saving *any* System setting (sound host, brightness, sleep) was silently overwriting `device.json` wholesale, wiping the `driver`/`dsp_host`/`dsp_port` keys that only `serve.py` reads at startup. `_persist_device()` now merges instead of overwriting.

### The Xilica console shows up
User: "ok we have a xilica console at 172.16.200.174" — a real Solaro QR1, reachable over the office VPN (same subnet as the office Pi).

- Confirmed reachable (TCP :10007 answers a `ping()`/`GETRAW`).
- **Committed the driver for real** — `xilica.py` had been sitting uncommitted for days while its own `test_xilica_driver.py` *was* committed; any fresh clone's test run was broken on a missing-module error the whole time. Fixed.
- **Cross-checked the driver against the real control-tag export + Solaro project XML.** Found `Zone{N}OverallGain`/`Zone{N}OverallMute` (a real per-zone master, at the "Audio Output Level" block) — `set_zone_level` had been faking it via crosspoints, and worse, `set_zone_mute` was reusing `ExclusionZone{N}`, the *same* tag `evac_suppress` uses for the safety EVAC silence-all. Confirmed from the project's own component wiring that these are genuinely separate DSP blocks in series, not aliases — fixed so an operator's everyday zone mute can't collide with the EVAC exclusion gate.

### "Swap the AHM for the Xilica" — nearly went badly, then didn't
Asked to swap the office Pi (which drives the REAL school's AHM) over to the Xilica. Checked first: the school's config has **8 zones**, the Solaro project only has **4** (`XilicaDriver.from_config` rejects any zone outside 1..4) — a straight swap would crash-loop the *live* site. Also: no USB audio interface/`sounddevice` installed on that Pi. Flagged both before touching anything; user chose to bench it on piface — then corrected to "just do it now on the office Pi," accepting the crash-loop.

Did it. Confirmed the crash exactly as predicted (`ValueError: zone 'zone_cblockint' maps to Solaro zone 5, outside 1..4`) — **site down, no bells, for real**. User chose to fix forward rather than revert. Turned out this was meant as a fresh proof-of-concept, not a forced fit onto the old 8-zone config — so:
- Backed up the real school's config → `config.school-backup-2026-07-20.json` on the Pi.
- Built a clean 4-zone POC config from scratch (`zone1..4` matching the Solaro's actual "Zone 1: Indoor"/"Zones 2-4: Outdoor"), zero scheduled bells so nothing auto-rings during testing.
- Driver flipped to `xilica`, `dsp_host=172.16.200.174` — came up clean.
- **Verified live, for real**: manual ring → read the Solaro's actual `Zone1BellsMute` state back over the wire → confirmed open on zone 1, muted on 2-4.

### USB audio interface (a Roland QUAD-CAPTURE was already plugged in)
Installed `libportaudio2` (system) + `sounddevice`/`soundfile`/`numpy` (venv — the `.[media]` extras-bracket syntax silently no-op'd over SSH, installed by name instead). Hit a real bug: `SoundDeviceBackend` hardcoded 48000 Hz; this interface's ALSA `hw:` device only accepts its own native 44100 Hz and rejected 48000 outright ("Invalid sample rate"). Fixed: falls back to the device's own reported default on failure — files are already resampled in software regardless, so any stream rate works.

**First real audible bell**: `School Bell.wav` (still in the media library from the old school setup) played out the interface while the Xilica correctly routed the DSP crosspoints — `media_warnings` clean, `status.playing` showed it live. Then wired up the remaining library files properly: 4 zones each got their own dedicated real test sound (Zone 1→All Clear.wav, Zone 2→Lock Down.wav, Zone 3→Lock In.wav, Zone 4→STOP.wav, each in its own single-zone group) + the 3 EVAC files as proper EVAC/Messages actions (Evacuate/On-Site/Off-Site, hold=True + a matching All Clear release) + `Paging Preamble.wav` set as the preamble chime. Live-tested the full EVAC hold→release cycle for real (`evac: true` → real audio → `evac: false`).

Also: the panel (`bell-panel`, 172.16.200.165) looked offline — turned out to be correctly asleep per its own wake/sleep policy (the config had picked up a real 12-bell timetable somewhere along the way, so the panel dutifully slept 30 min after the last bell). Widened `sleep_lag_min` to 600 for the rest of the test session, not a bug.

### "The sound library was not automatically loaded" → console was lying about the driver
Root cause: Sounds & Zones only ever branched Atlas-vs-everything-else, so the Xilica POC inherited AHM-branded copy throughout — worst offender, a "↺ Load from AHM library" button hitting a dead `/api/ahm/library` endpoint. The *real* library (the appliance's own uploads) was fine the whole time. Added a proper `isXilica` branch: correct column labels, AHM/Atlas pull buttons hidden (Xilica has no onboard library or `read_names`), and the "who plays the audio" badge/warning got Xilica-specific text — it has no internal player to fall back on at all, so "give it a slot" was actively wrong advice there.

### "Confirm this is working as per Xilica file design intention"
Traced the *actual* signal wiring from the project's `LA_LNK` (SRCID/DSTID) link data, not just control names:

```
Input Mixer → Zone Exclusion Mixer → Zone N Overall Ducker → Zone Matrix Mixer → … → Audio Output Level → Dante
```

Confirms `ExclusionZone{N}` sits early (cuts everything before ducking/output even sees it — right for the EVAC safety gate) and `Zone{N}OverallGain/Mute` sits right at the end (the true output master) — genuinely independent gates in series, both defaulting to FALSE/unmuted in the project's own shipped preset. Nailed the exclusion-polarity question that had been an open TODO since the driver was first scaffolded.

User then sent a screenshot of the actual Xilica Designer canvas — the designer's own note boxes confirmed it in plain English: *"[Exclusion Mixer] used to mute its outputs to not allow bells, paging through"*; *"[Input Mixer] used to choose which zone to send paging mic to and preamble"*; *"Paging Station mic mute to be triggered by push to talk button"*. All matched what was already built.

### "Use the project's ducking system"
The project has its own per-zone `Page & Alert Ducker` (fed by `Zone{N}AlertGain` + `Zone{N}PagingMicAlertGain`, ducks whatever else is playing in that zone). Wired live paging (`page_start`/`page_stop`) to engage this **additively** — alongside the ordinary crosspoint, not instead of it, since I couldn't fully verify from the XML alone whether the ducker passes its trigger signal through as audio or just uses it as a silent sidechain, and wasn't going to risk making a live announcer inaudible on an unverified assumption.

Verifying it against real hardware turned up **a genuine typo in the vendor's own project export**: zone 1's control is `Zone1PagingMicALertMute` (capital AL) — every other zone is spelled correctly. Confirmed by testing both spellings directly against the unit (correct spelling: no reply; the typo'd one: real data). Fixed the driver to match; wrote a systematic diff of every control name the driver constructs against the full 83-entry export to make sure there wasn't another one hiding — there wasn't.

### "With what we've learnt today, run a final bug check"
- Diffed all 71 constructed control names against the real export — 100% match.
- Found one real bug: `rebind_from_config()` (runs on every config edit) reset the ordinary crosspoint's stale-route cache but not the new ducking one — a zone remap could leave the ducking path skipping writes using a stale mapping. Fixed + regression test.
- Tightened `trigger_bell`'s docstring: it opens the Bells crosspoint for EVAC/message calls too, which is a harmless no-op for them specifically — that audio actually rides the appliance's separate "evac" USB channel into the Solaro's Alert input, which needs zero per-zone routing (`Zone{N}AlertGain` ships at unity on every zone by default) and — because it bypasses the Exclusion Mixer entirely — **can't be silenced by a zone exclusion at all**. Confirmed from the project's own preset defaults.
- Upgraded the exclusion-polarity comment from "almost certainly, confirm on the unit" to actually confirmed.

**260 tests pass.** Everything pushed to beta, deployed + verified on all three consoles (office Pi live on the Xilica POC, piface, docker demo — inert there since none of them select Xilica as their driver).

### Paging mic is Dante 5-6 — and a second real bug the bug check missed
User supplied the actual commissioning detail. Added a real `PagingSource` ("Paging Station Mic (Dante 5-6)") to the POC config via a live edit — and hit a second bug of the *exact same shape* as the one the final check had just fixed: `rebind_from_config()` recomputed zones/groups on a config edit but never `source_keys`, so a paging source added after the process started was invisible to the running driver (`page_start` → "no Solaro source mapping") until a full restart. Extracted a shared `_default_source_keys()` so `from_config`/`rebind_from_config` can't drift apart again — the systematic check caught one instance of "new state added without checking every place the analogous state gets reset," but not a second, pre-existing instance of the same class of bug that only live usage surfaced.

Verified live end-to-end after the fix: `/api/paging/start` correctly opened the ordinary crosspoint + the ducking path + the master mic mute together on the real unit (all three read back correctly over the wire); `/api/paging/stop` released all three cleanly. **Live paging is now fully proven on this hardware — not just at the driver-test level.** 261 tests, commit `283daf0`.

### Panel showed "not commissioned" → then the mic came up unmuted on boot
Live-tested paging on the actual wall panel. First problem: I'd added a second paging source while testing the auto-commission fix — the panel's auto-select only works with exactly one commissioned source, so it silently stopped offering paging. Removed the extra source; also learned the panel only picks its mic once at page load (not on its 2s poll), so a remote `/api/panels/reboot` was needed rather than waiting.

Then the real one: **user reported the mic came up unmuted right after the panel booted, before anyone pressed Talk** — a genuine safety gap. Root cause: the driver only ever writes mutes *reactively* (on page start/stop) and has no memory of the DSP's actual prior state. Checked the project's own preset: the master `PagingMicMute` control ships **unmuted by default**. Fixed: `connect()` (called once at appliance startup) now force-closes every commissioned paging source the same way `page_stop` does.

**Caught a real deadlock in the first version of that fix before it ever touched the office Pi.** It held the driver's lock across the whole muting loop, but everything underneath acquires that same non-reentrant lock too — the appliance's startup thread would have hung forever on every future restart. The unit test's fake driver didn't catch it (its `_send_line` stub skips locking entirely) — only testing against the real hardware did, as a background test that hung and had to be force-killed. Fixed, re-verified live twice (the hang, then the fix), deployed for real, and confirmed an actual service restart on the office Pi came up with the mic correctly muted. 263 tests, commit `9545d7b`.

### Inputs (Setup → Panel faders) were also silently dead
Added analog inputs named "Bells"/"Alert"/"Preamble" and asked where the volume was mapped to — nowhere, it turned out. `set_input_mute`/`set_input_level` had been reusing the paging-source dict instead of having their own mapping, so every Input silently no-op'd; AHM has a proper dedicated mapping for this, Xilica never got the equivalent. Fixed by matching an Input's name case-insensitively against the Solaro's four fixed named rows (Bells/Alert/Preamble/Spare — there's no generic numbered input bank the way the AHM has), which is exactly what the live commissioning had already done by naming them that way. Verified through the real `/api/input/level` endpoint the console fader calls — moved `BellAudioPlayerGain` and restored it. 267 tests, commit `33275df`.

### RTA meters added to the panel's Input faders
The Solaro exports a real per-source meter (`BellPlayerMeter` etc, -80..+40 dB) that nothing ever read — a `subscribe_meter()` method existed in the driver, but the UDP :10008 listener its own docstring promised was never built. Went with polling instead of building that infrastructure: a plain GETRAW through the same `_input_keys` mapping the faders already use — simpler, reuses the existing connection, plenty fast for a level meter. New `GET /api/inputs/meters` endpoint; the wall panel's External Inputs page now shows a slim red/amber/green RTA bar next to each fader (separate from the fader itself, which only shows the *set* gain, not the live signal), polling only while that page is actually showing. 271 tests, commit `4f0dc98`.

**Deploy mistake caught by verifying, not assumed**: `scp drivers/xilica.py service.py webapp.py pi@...:/home/pi/bellcommander/bellcommander/` — sources spanning subdirectories with one destination silently drops the `drivers/` path component, so `xilica.py` landed as a stray file next to `service.py` instead of overwriting the real one in `drivers/`. The office Pi restarted on the stale driver; `/api/inputs/meters` came back all `null` and looked exactly like a code bug. Caught by checking the deployed file actually changed (`wc -l`, `grep -c` for the new method) rather than trusting the restart succeeding — worth doing as standard practice after any deploy touching `drivers/`, given how much of today only got caught by exactly this kind of verification.

### EVAC now opens the paging mic hands-free
Requested: no push-to-talk should be needed in a real emergency — the mic should just be live the moment EVAC engages, and go back to muted on release. `evac_suppress(active)` now opens the paging mic (master, ordinary crosspoint, and the ducking tap, to every zone) when EVAC activates, and closes it again the same way `page_stop` does when it's released. Verified live with fresh-connection reads (ruling out same-socket staleness, which bit a couple of the checks along the way — same flakiness pattern as the rest of today, always settled correctly on a clean re-check).

**One open safety caveat I flagged rather than paper over**: the ordinary crosspoint this opens is itself downstream of the same Exclusion Mixer that EVAC engages — bells and paging share that gate, confirmed from the project's wiring — so unmuting the mic doesn't guarantee it's actually *heard* in an excluded zone via that path. The ducking-alert tap bypasses Exclusion and gets opened too, but whether it carries the mic's audio on its own (vs. being a silent trigger signal for the ducker) has never been confirmed by actually listening — only reasoned about from the block diagram. This is the one piece of today's Xilica work that's shipped but not fully hardware-verified: **needs someone to actually key up during a real EVAC test and confirm they're heard.** 273 tests, commit `48fa151`.

### Second DAC connected → split-device config broke live audio, real incident (21 Jul)
User plugged in a second USB DAC (HiBy R4). It didn't show on the Media page — confirmed a known PortAudio limitation, not a bug: hardware is enumerated once when the process starts, not scanned live. A fresh process saw it immediately; a restart of the service picked it up too. But by then the media config had been split across both devices (Bells/Preamble on the new DAC, BGM/EVAC on the original QUAD-CAPTURE) — **this actually failed live**: `media_warnings` reported the audio backend unavailable, and bells went genuinely silent. Direct testing confirmed the root cause: each DAC opens fine completely alone, but opening both together in one process throws "Invalid number of channels" deep in PortAudio's ALSA backend — a real driver-level limitation on this Pi, not something fixable in our code. Immediate fix: reverted the config back to all four roles on the one DAC that's been reliable all day, confirmed restored with a real bell ring.

While investigating, also found and fixed a genuine resource leak in the sample-rate fallback logic from earlier today: if the fallback attempt itself also failed after already opening one device successfully, nothing ever closed that stream — confirmed via `fuser` that the ALSA device sat "busy" for every later attempt, including the appliance's own next reconfigure, until a full restart. That's actually why a plain config save didn't recover it and a service restart was needed. Fixed with proper cleanup on that failure path too. Commit `f67e67a`.

**I made a real mistake here and want it on the record rather than glossed over**: while trying to reproduce the two-DAC conflict in an isolated script to verify my leak fix, I misidentified a process ID and ran `kill -9` on what turned out to be the *live* bellcommander service, not my test script — confirmed by the timestamps lining up exactly in the journal. Systemd's own restart-on-crash policy caught it and had the service back up within 3 seconds, and I verified it came back healthy immediately after. No lasting damage, but it was a genuine near-miss from being too casual about which PID I was targeting on a box running a live service, and worth remembering for next time: don't run exploratory kill commands against shared hardware on a machine that's also serving live traffic without being certain of the PID first.

### Restart appliance button added to System page
`POST /api/system/restart` — fires the actual `systemctl restart` after a short delay on a background thread so the HTTP response reaches the browser before the process dies (can't do this synchronously — you'd kill your own request handler mid-response). Console gets a danger-styled, confirm-gated "Restart appliance" button, deliberately last on the System page. Verified live: restarted in ~1 second, stayed stable. First test looked broken because I was *also* manually SSH-restarting in parallel to diagnose — three overlapping restarts in quick succession from testing too eagerly, not an actual bug. Added proper logging (was a bare `except: pass`) so next time doesn't need timestamp archaeology in `journalctl`. Commit `2ead599`.

While re-testing this, also caught the media config drifting back into a broken state — channels referencing outputs 0-3 on a 2-channel device (not the earlier two-device conflict, a new variant of the same class of problem from further hands-on experimenting with the Media Player page). Reverted to the known-good single-device config again, confirmed with a real ring.
