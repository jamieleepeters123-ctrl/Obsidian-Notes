# 2026-07-15 — Bell Panel: Beyond boot logo + kiosk white-flash fix

Continued the [[Beyond Bell Commander]] tablet wall panel. Goal today: make the boot look like a Beyond appliance, not a generic Linux tablet. Two wins — replaced the kernel penguin with the Beyond logo, and killed the white flash between the splash and the launcher.

## Replaced the boot penguin with the Beyond logo
The "penguin" during boot was **`logo.bmp` + `logo_kernel.bmp`** (identical 393×334, 24-bpp, top-down BMP) inside a Rockchip **RSCE resource** blob, itself inside a **signed U-Boot FIT** in the boot partition (`mmcblk0p3`).
- FIT is signed `sha256,rsa2048` with key-name-hint **`dev`** (the public rkbin dev key), **but** `androidboot.verifiedbootstate=orange` → bootloader unlocked, signature **not enforced**. So a modified boot image boots fine.
- **Safe in-place swap:** kept the new logo at the **exact same 394174-byte size** (no offsets move), overwrote both logo regions, updated the two RSCE per-entry **SHA1**s and the FIT `/images/resource` **SHA256**, left the stale rsa signature (ignored while orange). Kernel + device-tree left byte-identical → boot is rock-solid.
- **Gotcha:** the panel scans out in portrait (800×1280), so the logo renders **90° rotated** unless you pre-rotate the bitmap 90° CW. First attempt was sideways; fixed by baking in the rotation.
- Logo = the real blue **B** symbol + "Beyond" wordmark (scraped from the site) + "Bell Panel", on black so it blends with the boot screen.
- **Recovery net:** pristine boot dumped to `~/kiosk/boot.img.pristine` (one `dd` to revert over SSH); plus the full 4.97 GB `update.img` firmware + RKDevTool for a maskrom reflash. Flashed via `dd` to the partition; verified readback each time.

## The white flash (long fight — mostly fixed)
The flash between the Beyond splash and the launcher was **Chromium painting its window white for ~0.3 s on cold start** (GPU compositor clear, before first paint). This was a slog. What did **not** work: `--force-dark-mode`, `--default-background-color` (headless-only), `--disable-gpu` alone, and a `feh` cover held on top with an `xdotool windowraise` loop — that last one turned one flash into a **3-flash flicker war**, because with no window manager stacking is just map order and Chromium keeps re-raising its own window.

A big lesson: my on-device `xwd` capture rig (grab frames → whiteness timeline) is **too slow (~130 ms/frame) to catch a 0.3 s transient**, so it kept giving false "0 white frames" and I twice called it fixed when it wasn't. The only reliable test was slow-mo phone video.

**What actually worked — born off-screen:** launch Chromium chromeless (`--app`) with `--window-position` past the screen edge + occlusion/backgrounding disabled, so the window is created *off-screen*, paints the launcher where nothing shows, then `xdotool windowmove` slides it on once its `<title>` is set. The white physically can't reach the screen — structural, not a race.

**Where it landed:** the **big flash right after the Beyond logo is gone.** A much **shorter flash remains at the moment the window slides on-screen** (the move re-composite) — couldn't fully kill it without a window manager to enforce stacking. Added `--disable-gpu` + a brief cover to shrink it. Decided to **live with it for now**; the clean fix later is a tiny WM (openbox). **Bonus:** switching `--kiosk` → `--app` fixed the resolution/scaling issue too.

The bright screen ~5 s after the launcher is *not* the flash — it's the launcher auto-connecting to the saved server (`192.168.1.42:5000`, the light "PiFace" dashboard).

## The tablet is a paging station — built it (rev 0.2.0)
Turns out the panel's job is to be a **paging station**, replacing the customer's Q-SYS console (the Chiffley design: TSC-101-G3 + Core Nano). Those three UCI screens are the functional spec, and they're now built and running.

**The insight that made it small:** a page is the *same matrix operation as a bell* — route a source row to the target zones' columns — just from a mic's row instead of the playback row. The crosspoint work from July is doing all the heavy lifting. Which means **analog vs Dante is one config field**: it only picks which input range the channel number counts in, because both land on the same matrix. This site is analog (no Dante card); moving a site to Dante later is a two-field edit, no code, no retest.

**The interlock** is the bit worth remembering: a bell that falls due mid-announcement is *held* and rings when Talk is released (logged as rung, with the delay recorded), rather than truncating the thing people are actually listening to. Held past 60s it's dropped instead — by then a bell is misinformation. EVAC and an operator skip still outrank a page. **EVAC deliberately does not block paging** — talking to the school is what you most need during an evacuation.

**Watchdog:** the panel closes its own page on release/blur/tab-hide/pagehide, but if Chromium crashes or the tablet drops off the network *none* of those fire — open mic across the school, every bell silently held. So the tick loop closes any page older than 120s. That can't be the panel's job to guarantee.

**Panel** at `/panel` — standalone HTML/CSS/JS, not the React console (different device, has to cold-start fast on the ARM board). Four screens: Paging, EVAC, External Inputs, Zone Exclusions, deep-linked by `#hash`. EVAC renders whatever `config.messages` says through the existing `/api/message`, so the engine needed no new emergency code. Every emergency action is press-and-hold — a deliberate deviation from Q-SYS, where only Cancel is: that console sits on a desk, this panel sits in a corridor. Faders are hand-built divs; native vertical `<input type=range>` styles inconsistently across Chromium versions and gives a hairline track no fingertip can find.

**Installer-editable** (Setup → Panel): the panel always rendered 100% from config, but there was nowhere to *edit* that config — an installer would've been hand-editing JSON. Now: paging stations (incl. commissioning the mic), zone order + Inside/Outside + buttons-per-row, the macro rail, fader strips, limits. Nice reuse: a **Group is already "a named set of zones"**, which is exactly what a macro is — so Chiffley's "Oval" key is just one of the bell groups, no new concept. Considered Node-RED for this and argued against it: it'd be a passthrough middleman, its dashboard editor is a node graph + 12-column grid (not a Designer canvas), and "editable" would come to mean *editing flows* — N bespoke flow files instead of N validated configs. Node-RED still makes sense later for the tablet's relay/GPIO/RS-485 and site glue, against the engine's API.

**Trap avoided:** removing a zone from a station is `PagingSource.zones`, *not* deleting the zone — a zone is a real AHM channel bells ring through, so tidying a paging screen must never silently break the whole-school group. "Off this panel" (layout) and "excluded from paging" (policy) stay separate, and an excluded zone still shows on screen, greyed, rather than lying about the shape of the school.

**Constraint found: the AHM matrix is 16×16, so an AHM-16 caps at 16 zones. Chiffley has 19 — that site does not fit on one AHM.** Worth knowing before it's quoted as a Q-SYS replacement.

**Where it runs:** engine on the piface Pi (`192.168.1.42:8095`, systemd `bellcommander.service`, `BC_DRIVER=fakeahm` — the bench AHM at .168 was off); tablet kiosk repointed from `launcher.html` to `http://192.168.1.42:8095/panel` (backup `start-kiosk.sh.launcher-bak`, one-line revert). The `xdotool --name "Bell Panel"` lookup still matches "Beyond Bell Panel", so the white-flash fix is untouched. Demo copy rebuilt on the office Docker at **172.16.200.151:8090**. Tagged **v0.2.0** as a restore point (0.1.0 = the bell appliance; 0.2.0 = the paging station). 195 tests.
